# 运维与部署

## 固定版本

- Docker 与 Docker Compose
- Node 20
- pnpm 10.34.4
- Prisma 5.22.0
- MySQL 8.0
- Redis 7
- 固定 RELEASE 版本的 MinIO 与 MinIO Client

| 服务 | 端口 | 对外 |

| frontend | 8080 | 是 |
| backend | 3000 | 仅回环 |

## 环境变量

生产环境使用项目根目录 `.env`。首次部署从 `.env.example` 创建，所有 `CHANGE_ME` 必须替换。更新部署必须复制旧生产 `.env`，禁止用示例文件覆盖。

## 发布原则

1. 代码合并到 Git 主干或发布标签。
2. 服务器使用只读 Deploy Key 拉取 Git。
3. 生产 `.env`、`backups/` 和 Docker 命名卷只保留在服务器，不进入 Git。
4. 执行 `bash deploy-git.sh preflight` 和 `bash deploy-git.sh deploy`。
5. 部署脚本先构建镜像，再停写、备份 MySQL/MinIO、执行迁移、切换容器。
6. 验证健康检查、前端 `build-info.json` 与 Git 短提交一致。

`deploy-git.sh` 会在数据库同步前备份 MySQL、MinIO 和 `.env`。失败时自动保存诊断日志，默认只回滚代码；数据库和 MinIO 恢复必须显式执行 `CONFIRM_RESTORE=YES bash deploy-git.sh restore-data`。

## 数据同步

迁移容器使用本地 Prisma 5.22.0 执行正式迁移和幂等种子：

```text
node prisma/prepare-migrate.js
prisma migrate deploy
ts-node --transpile-only prisma/seed.ts
```

`prepare-migrate.js` 只处理从历史 `db push` 部署切换到 Prisma Migrate 的一次性基线标记：如果数据库已有核心业务表但没有 `_prisma_migrations`，会把 `20260703000000_init` 标记为已应用；新库则直接执行 baseline migration。

生产环境禁止使用 `prisma db push --accept-data-loss`。所有 Schema 变更必须先生成 `prisma/migrations/*/migration.sql`，经过评审和测试库验证后再发布。每次发布前必须确认 Schema 变更、数据库备份和回滚方案。

## 日常检查

- `bash deploy-git.sh status`
- 后端 `/api/v1/health`
- 前端 `/health`
- 前端 `/build-info.json`
- MySQL、Redis、MinIO 容器健康状态
- `backups/failures/` 是否存在新的故障记录

完整命令、旧包部署转换和数据保护步骤见根目录 `DEPLOYMENT.md` 与 `09_部署运维/Git推送部署与数据保护方案.md`。
